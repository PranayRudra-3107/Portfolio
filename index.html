About

I moved from Hyderabad to Germany in early 2025 to do my Master's in Applied Computer Science at TH Deggendorf. Before that I spent two years as a software developer — first at Aragen building .NET applications, then at Zelis on Angular-based healthcare software. Somewhere in between, I got serious about AI.

When something I build confuses people, I take it personally. So I try to understand the problem properly before touching the code — not because I read it in a book, but because I've shipped things that didn't land and had to figure out why.

Right now I'm deep into LLMs and AI systems — building tools that use satellite data, foundation models, and vector search to solve real regulatory and supply-chain problems. I'm also part of United AI e.V., a student AI collective at TH Deggendorf.

Design Philosophy

    Research before pixels

    I've learned the hard way that the obvious design usually solves the wrong problem. So I ask a lot of questions before opening Figma.

  • 02
    Constraints are creative fuel

    The best interfaces I've worked on came from tight timelines and real technical limits. Unlimited freedom tends to produce waffle.

  • 03
    Ship, measure, iterate

    I'd rather put something real in front of users and learn than spend three weeks polishing something nobody asked for.

Process

How I design
from problem to product.

It's not a straight line — more like a loop I keep refining. What changes per project is how much time I spend in each stage, not whether I skip one.

01

Discover

First thing I do is resist the urge to sketch anything. I talk to people, look at what already exists, and try to understand what the actual problem is — not just the one in the brief.

User Interviews Heuristic Evaluation Affinity Mapping
02

Define

I try to write the simplest possible problem statement from what I've found. If I can't say it in one sentence, I don't understand it well enough yet.

Personas User Journey Maps HMW Statements
03

Design

Rough sketches before Figma — always. I test with whoever I can get hold of. Real feedback from one confused person beats a beautiful screen nobody questioned.

Figma Component Libraries Usability Testing
04

Deliver

Having an engineering background helps — I can read the component library, know what's buildable, and write specs that developers don't need to translate into something else.

Design Tokens Angular / TypeScript Accessibility Audit
05

Measure

Shipping isn't the end. I look at support tickets, task times, drop-off points. The data usually tells a different story than the demo went well.

A/B Testing Analytics Retrospectives
