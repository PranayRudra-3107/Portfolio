Healthcare Benefits Dashboard — Redesign

Angular · TypeScript · Tailwind CSS

01 — Problem

A dashboard that created more questions than it answered

At Zelis, I was embedded in a cross-functional team responsible for a web portal used by thousands of employees to manage their healthcare benefits. The existing dashboard had been built incrementally over 4 years, accruing complexity with every new benefit type added.

Users regularly called support to answer questions the dashboard should have answered automatically: "What's my deductible?" "Which plan covers my dentist?" "How much have I spent this year?"

Support ticket analysis showed 60% of calls were information requests that could be self-served. The dashboard had the data — users just couldn't find it.

02 — Research

Understanding the mental model gap

I partnered with a product manager to run a mixed-methods research sprint over three weeks:

  • Support ticket analysis — categorised 200 recent tickets to identify the top 10 information needs
  • 5-second tests — participants described what they thought the dashboard showed them; accuracy was 40%
  • Card sorting — 12 participants sorted 30 benefit concepts to reveal their mental model
  • Contextual inquiry — observed 4 users attempting real tasks with screen-share

The core insight: users organised benefits by life event (doctor visit, prescription, dental checkup), not by benefit category (medical, pharmacy, dental). The dashboard was structured by category. They were speaking different languages.

Card sorting results — users grouped by life event, not benefit category
03 — Define

Reframing the information architecture

Working with the product manager and engineering lead, I established three design principles for the redesign:

  • Answer the question before it's asked — surface "remaining deductible" and "last visit" in the first viewport
  • Life-event navigation — restructure secondary navigation around common scenarios, not benefit types
  • Plain language first — replace insurance jargon with plain English, with technical terms available on demand
04 — Design

Restructuring without rebuilding from scratch

Given engineering constraints (existing Angular component library, PostgreSQL data model), a full rebuild wasn't feasible. I designed a restructuring: same data, completely rethought presentation layer.

Key design decisions:

  • Benefit summary cards — three prominently-placed cards (Medical, Dental, Vision) with the three most-asked metrics each, immediately visible on load
  • Spend timeline — a simple bar chart replacing a raw data table for YTD expenditure
  • Smart search — a "Find care" bar that accepted natural language ("Does my plan cover physiotherapy?")
  • Deductible progress bar — a prominent visual tracker replacing a number that required mental calculation
Before / After — homepage: 14 cards → 3 summary cards + contextual detail

I delivered Figma specs with annotated responsive breakpoints and a component inventory to the Angular team. My own engineering background helped bridge the gap — I could speak to implementation constraints directly in design reviews.

05 — Testing

Validation and iteration

Two rounds of usability testing with 5 participants each (recruited from existing user base with coordinator support):

  • Round 1: Moderate success — users found deductible info 80% faster, but the plan-comparison flow still confused 3 of 5 participants
  • Round 2 (after plan-comparison redesign): 95% task completion on top 5 tasks; qualitative feedback was markedly more positive

We introduced a structured code review process for UI components that I championed — this reduced regression bugs in the design system by catching inconsistencies earlier.

06 — Outcome

Measurable results

  • Support tickets dropped 28% within 60 days of launch
  • CSAT score improved from 4.2 to 4.7 over the following quarter
  • Time-on-task for "find my deductible status" fell from 3 min 40 sec to 28 sec in follow-up testing
  • The component library we built during this project was adopted as the team's design system baseline

The most impactful change wasn't visual — it was restructuring the information architecture to match users' mental models. Design and engineering alignment made it possible to ship without breaking the existing data infrastructure.
