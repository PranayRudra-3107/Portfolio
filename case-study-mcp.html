MCP Server Control Panel

FastAPI · Python · Developer tooling

01 — Problem

Developer tools aren't exempt from UX debt

The Model Context Protocol (MCP) defines how AI applications connect to context sources — databases, APIs, documents. The prototype MCP server I built with FastAPI had all the right capabilities: authentication, context-source management, traceability logs, LLM tool integration.

But the management interface was essentially a Swagger UI wrapper. Onboarding a new context source required reading the docs, hand-crafting JSON payloads, and debugging authentication errors with no helpful feedback. Even experienced developers were making errors on first setup.

Good developer UX isn't about making things simpler — it's about making complex things learnable, and learnable things fast.

02 — Research

What developers actually need

I interviewed 4 developers who had used or attempted to use the server during a university AI workshop. Key friction points:

  • Authentication setup was the #1 drop-off point — users didn't know which credential format to use
  • Context source status was not visible — users had to make a test API call to verify a source was live
  • Error messages were technical but not actionable — "422 Unprocessable Entity" told them what failed, not how to fix it
  • No activity log — traceability existed in the backend but wasn't surfaced in the UI

I also benchmarked against Supabase, Railway, and ngrok dashboards as references for developer tool UX done well.

Benchmark analysis — Supabase, Railway, ngrok: patterns for status visibility + progressive disclosure
03 — Define

Design goals for developer tooling

  • Guided setup — step-by-step wizard for first-time context source configuration, not a blank form
  • Live status at a glance — every connected source shows health status with last-checked timestamp
  • Actionable errors — error states include a "What to do" suggestion, not just an error code
  • Trace visibility — a paginated activity log with filterable event types (auth events, queries, errors)
04 — Design

Designing for two modes: setup and operation

The interface separates two distinct user states: first-time setup (high guidance needed) and ongoing operation (speed and visibility needed). A step-indicator wizard handles setup; a compact dashboard handles operation.

Key design decisions:

  • Connection wizard — 4-step flow (source type → credentials → test connection → confirm) with inline validation and copy-paste credential snippets
  • Status chips — green/amber/red status badges on each source card, with tooltip showing last successful query time
  • Smart error recovery — the auth error state includes a collapsible "Common causes" section matched to the specific error type
  • Activity log — filterable by event type, searchable by source name, with JSON payload expansion on click
Connection wizard — step 2 of 4: credential entry with format hints + live validation
05 — Testing

Validation with real developers

I ran a task-based evaluation with 3 developers who had not used the system before:

  • Task: "Connect a PostgreSQL database as a context source and verify it's live"
  • All 3 completed successfully on first attempt (vs 1 of 3 with the original Swagger UI)
  • Mean time reduced from 14 minutes to 5.5 minutes
  • Zero critical auth errors in the wizard flow (previously the most common failure)

Iteration after testing: added a "copy to clipboard" button for auto-generated API keys — a simple affordance that 2 of 3 participants had manually done anyway.

06 — Outcome

Results

  • Context-source onboarding time dropped ~60%
  • Authentication error rate in test scenarios: 0 critical failures
  • Traceability features enabled faster debugging — the activity log became the first stop for diagnosing unexpected LLM behaviour
  • The design patterns established here directly informed the team's internal tooling conventions

Developer experience design taught me that the best interfaces are those that make the right path the obvious path — not by hiding complexity, but by sequencing it.
